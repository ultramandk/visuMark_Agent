\section{应用背景与任务定义}
\subsection{应用背景}
随着互联网的快速发展，网页的样式和信息量都在经历日新月异地增长，为人们完成特定的网页任务带来了挑战。
在多模态大模型技术成熟的背景下，基于VLM/LLM技术实现的多模态网页Agent能够实现辅助和某种程度地替代人类
完成对应的网页任务，具有广阔的应用前景。Web Agent可以作为自动化测试工具，辅助软件开发团队进行前端的
冒烟测试与回归测试。Web Agent也可以作为智能个人助理，帮助个人用户执行复杂任务，降低网页任务的操作成本。
除此以外，Web Agent可以实现复杂数据的收集与分析、改进针对障碍人士的辅助功能等任务。

\subsection{任务定义}
我们计划实现一个能够理解自然语言指令，并通过浏览器自主完成复杂任务的智能代理。
用户将网页任务目标作为输入，在执行任务的每一步中
获取网页的 Accessibility Tree并根据网页截图进行SoM标记，
将网页截图与SoM输入多模态大模型，生成对应的指令，并使用playwright执行并等待页面反馈、进行任务的下一步。
基于任务导向，我们计划实现：
\begin{itemize}
    \item 一个SoM内容标注器，根据网页截图和Accessibility Tree生成元素边界框和对应的id，提高模型的准确率
    \item 一个基于VLM的Web Agent，根据用户输入进行任务的分步计划、在每一步根据网页截图信息和SoM标注生成对应的网页操作指令
    \item 一个playwright执行器，解析Agent输出的执行指令并执行，刷新网页进行下一步任务
\end{itemize}


\section{相关工作调研}
本课题相关工作涉及多模态GUI代理、网页自动化框架、任务规划与视觉定位等方向，以下列举代表性论文与开源项目。

\subsection{多模态网页Agent}
\begin{itemize}
    \item \textbf{WebVoyager} \cite{webvoyager}: 提出基于GPT-4V的端到端网页Agent，在真实网站上通过截图和HTML辅助实现高成功率，验证了大视觉模型在网页导航中的可行性。
    \item \textbf{Fara-7B} (Microsoft Research) \cite{fara7b}: 一个7B参数的多模态模型，结合Magentic-UI框架，使用Playwright执行操作，在控制复杂度的同时实现良好性能，是本课题的核心参考。
    \item \textbf{WebAgent} 综述 \cite{webagent_survey}: 对基于LLM的网页自动化Agent进行了系统分类，总结了规划、感知、执行的典型架构。
\end{itemize}

\subsection{视觉定位与混合感知}
\begin{itemize}
    \item \textbf{SoM (Set-of-Mark)} \cite{som}: 通过为可交互元素叠加数字标记，将 GUI 定位任务转化为标记选择任务，显著提升 VLM 在界面上的 grounding 准确率。本系统将 SoM 作为预处理模块，为 VLLM 提供可直接引用的标记索引。
    \item \textbf{R-VLM} \cite{rvlm}: 提出IoU感知的目标函数和缩放区域候选框，将GUI元素定位精度大幅提升，在Mind2Web基准上获得高准确率，为改进视觉定位提供了理论支持。
    \item \textbf{WebGUM} \cite{webgum}: 证明同时利用HTML/可访问性树和视觉信息比纯视觉或纯DOM方法更有效，为本课题混合感知策略提供了实验依据。
\end{itemize}

\subsection{相关开源项目}
\begin{itemize}
    \item \textbf{Playwright} \cite{playwright}: 现代浏览器自动化框架，提供跨浏览器、可靠的选择器和等待机制，本课题的执行层核心。
    \item \textbf{OpenClaw / Nanobot} \cite{openclaw}: 开源的多模态Agent实现，展示了从截图到操作的简易流水线，但缺乏精心设计的错误恢复和混合感知。
    \item \textbf{SMART-LLM} \cite{smartllm}: 多Agent协作框架，用于复杂任务规划，可借鉴其任务分解与验证机制。
\end{itemize}

\section{系统方案设计}

\subsection{SoM视觉输入模式}

SoM（Set-of-Mark）模式是系统的主要运行模式。该模式将网页截图与可交互元素的视觉标记相结合，使VLM能够直观地看到页面上的可操作元素，并通过引用标记编号来精确指定操作目标。其整体架构如图\ref{fig:arch_som}所示，遵循ReAct（Reasoning + Acting）范式，每步操作分为感知（Perceive）、推理（Reason）、执行（Act）和验证（Verify）四个阶段。

\begin{figure}[!h]
    \centering
    \includegraphics[width=0.9\textwidth]{arch_som.png}
    \caption{SoM视觉输入模式的系统架构}
    \label{fig:arch_som}
\end{figure}

\subsubsection{感知阶段（Perceive）}

感知阶段负责将当前网页的视觉状态转化为VLM可以理解的结构化表示。该阶段的输入为Playwright控制的浏览器页面，输出为SoM标注后的截图以及可交互元素列表。

\paragraph{ElementExtractor — 可交互元素提取}

\texttt{ElementExtractor}模块通过CSS选择器扫描页面DOM，提取所有可交互元素。主选择器覆盖了HTML原生交互标签（\texttt{<button>}、\texttt{<a>}、\texttt{<input>}、\texttt{<select>}、\texttt{<textarea>}）、WAI-ARIA角色属性（\texttt{[role="button"]}等）、事件处理器标记（\texttt{[onclick]}、\texttt{[tabindex]}）、无障碍标签（\texttt{[aria-label]}）以及QQ邮箱自定义的无障碍属性（\texttt{[data-a11y]}）。

\begin{lstlisting}[language=Python, caption=INTERACTIVE\_SELECTOR定义, label=lst:selector]
INTERACTIVE_SELECTOR = (
    "button, a, input, select, textarea, "
    "[role='button'], [role='link'], [role='checkbox'], "
    "[role='radio'], [role='combobox'], [role='listbox'], "
    "[role='menu'], [role='tab'], [role='switch'], "
    "[onclick], [tabindex], "
    "[aria-label], [data-a11y]"
)
\end{lstlisting}

对于采用React、Vue等前端框架构建的SPA页面（如新版QQ邮箱），按钮元素通常为纯\texttt{<div>}或\texttt{<span>}标签，不含任何上述可匹配属性。此时提取模块会遍历页面所有元素，检测其计算样式中\texttt{cursor}属性是否为\texttt{pointer}，并结合可见文本长度（1--8字符）识别为框架按钮。这一策略能够有效捕获纯div按钮。

所有候选元素经过以下过滤后，按照视觉位置排序并分配编号：
\begin{itemize}
    \item \textbf{可见性过滤}：排除\texttt{display:none}、\texttt{visibility:hidden}、\texttt{opacity:0}的元素，这些元素渲染后在视觉上不可见。
    \item \textbf{尺寸过滤}：排除面积小于4px²的微小元素。
    \item \textbf{交互性过滤}：排除\texttt{disabled}属性和\texttt{pointer-events:none}样式的不可交互元素。
\end{itemize}

滤除这些明显不可操作的元素，能够大大减少som标注图片中的视觉干扰，提升VLM的定位准确率。

\paragraph{SoMMarker — 视觉标记绘制}

\texttt{SoMMarker}模块使用PIL（Python Imaging Library）在页面截图上绘制交互元素的视觉标记。为每个候选元素绘制指定颜色的矩形边框，并在边框上方或内侧放置带纯色背景的数字标签。标签采用12种高对比度颜色循环分配，相邻元素尽量使用不同颜色以便于区分。当标签因元素密集而可能重叠时，模块自动将标签移至下一个非重叠候选位置，确保所有标签清晰可读。

例如，bing主页的SoM标记图如图\ref{fig:som_eg}所示：

\begin{figure}[H]
    \centering
    \includegraphics[width=0.8\textwidth]{som_eg.png}
    \caption{SoM标记图示例}
    \label{fig:som_eg}
\end{figure}

\paragraph{DOMBridge — 编号到DOM的双向映射}

\texttt{DOMBridge}维护SoM标记编号与DOM元素之间的双向映射关系。每个标记编号被映射至三个关键信息：元素的CSS选择器（用于Playwright定位）、元素的边界框（用于坐标回退点击）以及后端节点ID（用于Mind2Web评测）。在执行阶段，通过\texttt{get\_bbox(id)}可获取指定编号元素的边界框，\texttt{som\_id\_to\_selector(id)}可将编号转换为Playwright定位器，从而能够执行点击或输入操作。

\subsubsection{推理阶段（Reason）}

推理阶段将感知阶段的结构化输出组装为VLM可理解的prompt，调用模型获取下一步操作决策。

\paragraph{Prompt组装}

\texttt{build\_som\_user\_prompt}函数负责组装user prompt，其结构包含以下部分：
\begin{itemize}
    \item \textbf{用户任务描述}：用户的自然语言任务。
    \item \textbf{当前页面信息}：页面URL与标题。
    \item \textbf{当前计划}：从上一步推理输出中提取，帮助模型保持任务连贯性。
    \item \textbf{历史操作记录}：最近5步，以\texttt{[OK]}/\texttt{[NO EFFECT]}/\texttt{[FAIL]}前缀区分执行结果。
    \item \textbf{可交互元素文字列表}：每行格式为\texttt{[编号] <标签> "文本内容" aria="无障碍标签"}，供模型与截图中的视觉标记交叉参照。
\end{itemize}
完整的prompt由上述user prompt与system prompt（包含角色设定、响应格式、可用动作表、规则）拼接而成。

\paragraph{登录与验证码的人机协同机制}

现今大量网页需要进行登录或验证才能体验完整功能，而agent本身难以独立完成验证码识别和登录流程，因此一定场景下，人工的介入是有必要的。

系统采用"自动检测 + VLM视觉判断"双层策略应对登录和验证码场景。程序化检测层在感知阶段运行，通过扫描页面DOM中的密码输入框、扫码登录关键词（"微信登录""QQ登录""二维码登录"等），快速识别登录页面。检测到后，Agent直接暂停并通知前端等待人工介入。

VLM视觉判断层作为补充，当VLM在截图或元素列表中识别到验证码、登录表单或扫码界面时，输出\texttt{captcha}或\texttt{login}操作。这两种操作被定义为终端操作，执行后跳过验证阶段，直接进入暂停等待。用户完成登录或验证，点击继续后，Agent重新感知页面并恢复任务循环。

\subsubsection{执行阶段（Act）}

执行阶段将VLM的决策转化为浏览器实际操作，并处理操作后的页面状态变化。

\paragraph{ActionExecutor — 多层回退点击}

\texttt{ActionExecutor}负责执行VLM输出的操作指令。

例如对于CLICK操作，采用3层回退机制以确保最大的执行可靠性：

\begin{enumerate}
    \item \textbf{Playwright点击}：产生浏览器原生事件，适用于大多数标准元素。
    \item \textbf{JS MouseEvent派发}：通过\texttt{dispatchEvent}派发完整的mousedown/mouseup/click事件序列。当Playwright点击无效时，此方法可绕过部分元素的事件拦截。
    \item \textbf{坐标回退}：通过获取元素坐标，执行Playwright屏幕坐标点击。
\end{enumerate}

对于TYPE操作，执行器强制要求先通过JS \texttt{focus()}和\texttt{select()}将焦点赋予目标输入框，清除已有内容后再通过Playwright键盘输入新文本。

\paragraph{新标签页检测与同步}

执行CLICK或PRESS操作后，系统检测浏览器上下文中的页面数量变化。若检测到新打开的标签页，自动将当前操作页面切换至最新标签页，同时更新截图和前端显示，确保Agent始终在最新的页面上下文中进行下一步操作。

\subsubsection{验证阶段（Verify）}

验证阶段在执行完成后立即对操作效果进行评估，为下一轮推理提供反馈信息。

\paragraph{全图像素对比}

通过PIL的\texttt{ImageChops.difference}对操作前后的页面截图进行逐像素相减，统计差异像素比例。若差异像素少于$0.05\%$，直接判定操作无效，跳过后续验证步骤。

\paragraph{DOM错误检测}

在像素对比之后（无论其是否触发），通过JavaScript扫描页面DOM中的错误提示元素。检测目标包括：WAI-ARIA标准角色（\texttt{[role="alert"]}等）、通用错误toast和表单验证消息。该方法利用错误提示必然出现在DOM中的特性，以极低成本覆盖了VLM可能遗漏的错误弹窗场景。

\paragraph{验证反馈与历史记录}

验证结果（包括观察描述、是否应当重试、回退操作、替代操作）被记录在\texttt{StepRecord.verification}字段中。下一轮推理时，\texttt{build\_som\_user\_prompt}将最近步骤的验证结果格式化为\texttt{[OK]}（验证通过）、\texttt{[NO EFFECT]}（操作无效）或\texttt{[FAIL]}（执行失败）标记，连同失败原因一起展示在历史记录中。当连续出现2次以上失败标记时，prompt中额外添加警告信息，引导VLM尝试完全不同的操作策略。

\subsubsection{前端Web交互界面}

前端基于FastAPI WebSocket与纯HTML/CSS/JavaScript构建，核心功能包括任务配置、Agent步骤实时展示、浏览器画面实时流和键鼠交互反馈。

\paragraph{实时截图流（CDP Screencast）}

前端通过独立WebSocket通道接收浏览器实时画面流。后端利用Playwright的CDP接口启动\texttt{Page.startScreencast}，以960×540分辨率、约15fps的频率持续截取当前标签页画面，通过专用WebSocket推送至前端。该通道与控制消息WebSocket完全分离，避免了高频帧数据阻塞控制消息的传输。

Screencast模块在浏览器切换标签页时自动检测并重建CDP会话——当帧超时或会话断开时，通过\texttt{env.get\_cdp\_session()}重新获取新标签页的CDP连接并重启推流，确保实时画面始终跟随Agent当前操作的标签页。

\paragraph{键鼠交互反馈}

前端的实时显示窗口可以直接接收用户的键盘和鼠标事件（click、wheel、keydown等），将事件类型与坐标发送至后端。后端通过CDP的键鼠操作等接口将用户操作转发至浏览器，实现用户在Web界面上直接操控Agent浏览器窗口的能力。

\section{实验评测与分析}
针对项目模型的SoM输入与HTML输入模式分别设计了评测实验。所有本地模型均在一台RTX 3090 24GB $\times 1$，内存60GB的计算机上运行，并使用Qwen 3VL-8B-Instruct作为本地测试模型,使用API调用Qwen3.7-max测试。
选取任务成功率(SR)、元素准确率(Ele. Acc)、任务单步操作的F1分数(OP.F1)、任务单步成功率(Step SR)以及单步平均耗时作为评价Web Agent性能的标准。
其中，Ele. Acc表示各任务内选择正确网页元素的步数比例跨任务的均值，Op.F1分数表示单步操作的F1值，Step SR表示各任务内元素与操作均正确的步数比例跨任务的均值，SR表示所有步骤均成功执行的任务数占数据集总任务数的比例。

\begin{align*}
    \text{Ele.Acc} &= \frac{1}{M} \sum_{i=1}^{M} \frac{1}{N_i} \sum_{j=1}^{N_i} \mathbb{I}(\hat{e}_{i,j} \in S_{acc}(e_{i,j})) \\[6pt]
    \text{Op.F1}_{i,j} &= \mathbb{I}\bigl(\hat{op}_{i,j} \in S_{op}(op_{i,j})\bigr) \times
                         \begin{cases}
                            1 & op_{i,j} = \text{CLICK} \\[4pt]
                            \text{F1}(v_{i,j}, \hat{v}_{i,j}) & op_{i,j} \in \{\text{TYPE}, \text{SELECT}\}
                         \end{cases} \\[4pt]
    \text{Step SR} &= \frac{1}{M} \sum_{i=1}^{M} \frac{1}{N_i} \sum_{j=1}^{N_i} \Bigl[ \mathbb{I}(\hat{e}_{i,j} \in S_{acc}(e_{i,j})) \times \mathbb{I}(\text{Op.F1}_{i,j} = 1) \Bigr] \\[6pt]
    \text{SR}_i &= \prod_{j=1}^{N_i} \Bigl[ \mathbb{I}(\hat{e}_{i,j} \in S_{acc}(e_{i,j})) \times \mathbb{I}(\text{Op.F1}_{i,j} = 1) \Bigr]
\end{align*}
其中$M$为数据集内的总任务数，$N_{i}$为第$i$个任务的步骤数，$e_{i,j},\hat{e}_{i,j}$分别为第$i$个任务第$j$步中Ground Truth的等价网页元素集合和Agent预测的网页元素，$op_{i,j} \in \{\text{CLICK},\text{TYPE},\text{SELECT}\}$为Ground Truth操作类型，$\hat{op}_{i,j}$为预测操作类型，$v_{i,j},\hat{v}_{i,j}$分别为Ground Truth和预测的操作值。
$S_{op}(\text{CLICK})=\{\text{CLICK}, \text{HOVER}, \text{PRESS}\}$为Mind2Web中将$\text{HOVER}$和$\text{PRESS\_ENTER}$映射至$\text{CLICK}$的等价操作集合；$\text{F1}(\cdot,\cdot)$为F1分数。

\section{错误分析与AGI局限性讨论}

前文的实验评测揭示了当前VLM驱动的Web Agent在真实网页场景中的多项性能瓶颈。本节通过对典型错误案例的深入分析，建立系统性的错误分类体系，并在此基础上讨论当前AGI技术在Web Agent领域的能力边界与根本性局限。

\subsection{典型错误案例分析}

为深入理解Agent的失败模式，我们从Mind2Web数据集评测和自建SoM测试集中选取了具有代表性的错误案例进行分析。每个案例展示了Agent在某类典型场景下的失败过程与根因。

\subsubsection{案例一：密集布局下的元素定位错误}

\textbf{任务场景}：在B站首页的分区选择页面中，Agent需要点击"影视"分区按钮进入对应分区。该页面分类标签密集排列，且页面中央同时弹出了活动推广弹窗，弹窗的半透明边框与分区按钮的SoM标记在视觉上高度重叠。

\textbf{Agent行为}：VLM在SoM标注截图中将第\#91号元素（影视分区按钮）与第\#57号元素（弹窗的半透明边框区域）混淆，错误地输出了 \texttt{CLICK \#57}。点击弹窗边框后页面未跳转至影视分区，Agent继续在错误页面上执行后续操作，最终任务失败。

\textbf{根因分析}：在密集、重复的UI布局中，SoM标记的边界框可能部分重叠（特别是当元素间距小于8px时），导致VLM无法准确区分相邻编号。此外，相邻元素的文字标签无法提供有效的语义区分，VLM仅凭视觉位置信息难以做出精确判断。该问题在Mind2Web Cross-Task评测中尤为突出——Qwen3-VL-8B的元素准确率仅41.9\%，意味着近六成的步骤中模型选择了错误的页面元素。

\subsubsection{案例二：操作执行失败——SoM标记与实际DOM元素不一致}

\textbf{任务场景}：在某饰品交易网站首页，Agent需要点击"饰品市场"入口按钮进入商品列表页。SoM标注将该按钮标为\#2号元素。

\textbf{Agent行为}：VLM正确识别了目标按钮（输出 \texttt{CLICK \#2}），Playwright原生点击和JS MouseEvent派发均未能触发页面跳转，href导航因该按钮为\texttt{<div>}标签（非\texttt{<a>}标签）而无法提取链接，坐标回退点击也因页面动态加载导致元素位置偏移而失败。4层回退机制全部耗尽后，Agent在下一步中重复尝试同一元素，陷入循环直至超时。

\textbf{根因分析}：该案例暴露了两层问题。第一层是执行层面：现代前端框架中的\texttt{<div>}按钮缺乏原生交互语义，其点击行为完全依赖框架的事件委托机制——仅在特定DOM子树中监听事件并内部路由。即使Playwright触发了完整的MouseEvent序列，框架的路由逻辑可能因事件\texttt{isTrusted}属性、事件传播路径差异或组件内部状态而拒绝响应。第二层是Agent推理层面：Agent缺乏对"点击后页面应如何变化"的预期模型，因此在操作无响应时无法判断是"元素不对""方式不对"还是"页面卡顿需等待"，只能盲目重试。该问题对应错误分类体系中的第8类（点击/输入执行失败），估计占总失败步骤的约10\%。

\subsubsection{案例三：操作值不完整——文本截断与格式偏差}

\textbf{任务场景}：在邮件编辑页面，Agent需要向收件人字段输入邮箱地址 \texttt{"739862481@qq.com"}。

\textbf{Agent行为}：VLM输出了 \texttt{TYPE \#2 "739862481@qq.co端午安康"}，邮箱地址的域名部分被截断，同时将邮件正文的主题文本错误地写入了收件人字段。

\textbf{根因分析}：VLM在处理长文本任务描述时存在"注意力衰减"现象——模型倾向于提取任务描述的前几个词作为操作值，而忽略后续的关键限定词。在Mind2Web的TYPE操作评测中，实际操作值的token-level F1分数反映了这一问题的严重程度：操作值越长、信息密度越高，模型输出的完整性就越差。

\subsubsection{案例四：操作无效——页面状态已达目标但Agent继续操作}

\textbf{任务场景}：在比较茅台与腾讯股价涨幅的任务中，Agent已成功打开茅台股价信息页面，但未能识别到当前页面即为目标页面，继续执行多余的搜索操作。

\textbf{Agent行为}：VLM未能识别到答案已在当前页面可见（搜索框下方已展示茅台股价卡片），而是继续滚动页面、点击无关元素，直至消耗全部30步配额后以失败告终。验证阶段的像素对比虽检测到截图变化，但均为无效的滚动和探索性操作。

\textbf{根因分析}：该案例暴露了VLM在"答案识别"与"任务终止判断"上的核心弱点。尽管System Prompt明确要求"一旦看到答案立即输出answer"，VLM在实际推理中容易陷入"继续探索"的行为模式，尤其在于：（1）页面信息密度高、目标答案在视觉上不够突出时；（2）任务描述中包含多个子目标时，模型可能完成了一个子目标后继续探索其他子目标；（3）VLM缺乏对"已获取足够信息"的元认知判断能力。

\subsubsection{案例五：跨域泛化失败——HTML模式下模型对新网站布局无知}

\textbf{任务场景}：Mind2Web Cross-Domain评测中，Agent需要在训练数据中未见过的网站类型上完成任务。

\textbf{Agent行为}：以Qwen3-VL-8B为例，在Cross-Task上的Step SR为33.9\%，但在Cross-Domain上降至28.0\%。面对全新的网站布局和DOM结构，模型在候选元素排序（BERT阶段）表现下降，即便候选列表包含正确元素，VLM也倾向于选择网站中更"常见"的非目标元素（如导航栏中的链接、搜索框）。

\textbf{根因分析}：BERT排序器（all-MiniLM-L6-v2）的嵌入空间对通用语义敏感但对特定网站的DOM结构语义不敏感。当新网站的HTML结构与训练分布差异较大时，正确元素的候选排名可能靠后，影响VLM的最终选择。同时，VLM本身对新类型网页的"界面直觉"较弱，在缺乏足够的上下文示例时，偏向于选择高可见度、高频率的交互元素（如页面顶部的导航链接）。

    & 5. 操作类型错误 & CLICK/TYPE/SELECT混淆或冗余步骤 & $\sim$12\% \\

    & 7. 任务终止误判 & 过早Answer（信息不完整）或过晚Answer（额外探索） & $\sim$8\% \\
\midrule
\multirow{2}{*}{执行层}
    & 8. 点击/输入执行失败 & 4层回退全部耗尽：元素不可见、iframe限制、反爬拦截 & $\sim$10\% \\
    & 9. 页面跳转异常 & 新标签页未正确切换、URL导航超时 & $\sim$5\% \\
\midrule
验证层 & 10. 验证误判 & 像素对比假阳性（广告轮播等动态内容）或假阴性（微小但有效的DOM变化） & $\sim$4\% \\
\bottomrule
\end{tabular}
\end{table}

\textbf{感知层错误}发生于Agent对网页状态进行建模之前，直接影响后续推理的输入质量。元素漏检主要源于现代前端框架（React、Vue）中大量使用无语义的\texttt{<div>}/\texttt{<span>}标签替代原生HTML交互元素，CSS选择器和启发式\texttt{cursor:pointer}检测虽能覆盖大部分场景，但对完全自定义的事件委托模式（如在整个容器上监听click事件并内部路由）仍存在漏检。SoM标记冲突在元素密度超过每平方厘米3个时显著增加，此时标签文字和边框的重叠使得VLM无法可靠地分辨相邻元素。

\textbf{推理层错误}是占比最高的错误来源（合计约65\%），集中体现了当前VLM在网页任务理解上的核心瓶颈。元素选择错误（约35\%）是所有错误中占比最高的单一类别：Mind2Web Cross-Task上Qwen3-VL-8B仅41.9\%的元素准确率即说明近六成步骤中模型未能选择正确元素。操作类型与操作值的偏差进一步降低了Step SR至33.9\%。任务终止误判在SoM模式下尤其突出——模型在复杂多步任务中缺乏对"任务完成度"的准确评估能力。

\textbf{执行层错误}发生于VLM已做出正确决策但浏览器操作无法完成的场景。值得注意的是，本系统的4层点击回退机制（Playwright原生→JS事件派发→href导航→坐标点击）已覆盖大多数执行异常，使执行层错误占比控制在约15\%。但在跨域iframe（如QQ邮箱的撰写页面嵌入独立域的iframe）和激进的反爬机制（如Bing的\texttt{preventDefault()}拦截）场景下，执行仍可能失败。

\textbf{验证层错误}占比最低（约4\%），主要挑战在于区分"有效的页面变化"与"无意义的动态内容"。广告轮播、实时数据更新、动画效果均会导致像素级变化，但并非Agent操作的结果。DOM错误检测作为补充机制能覆盖大部分弹窗错误，但对非DOM形式的错误提示（如Canvas渲染的toast）无法捕获。

\subsection{AGI局限性讨论}

上述错误分析不仅揭示了当前Web Agent系统的工程改进方向，更指向了当前AGI（通用人工智能）技术在真实场景中面临的若干根本性局限。本小节从六个维度进行系统讨论。

\subsubsection{视觉理解 $\neq$ 语义理解}

当前VLM（包括表现最好的Qwen3.7-max）对网页截图的理解本质上是模式匹配而非语义理解。模型通过大规模预训练学习了像素模式与文本之间的统计关联，但缺乏对UI元素的"功能性理解"。具体表现为：模型可以在一张包含搜索按钮的截图中识别出"这是一个搜索按钮"，但该识别依赖于视觉特征（按钮形状、颜色、文字标签）与训练数据中相似模式的匹配。当面对全新的、与训练分布差异较大的UI设计时（如非标准控件、定制化组件），模型的识别准确率显著下降。这种"见多识广但未真正理解"的特性是当前视觉语言模型的根本局限。

证据来自实验数据：SoM模式下任务成功率$73.3\%$相比HTML模式下$40.0\%$（纯文本，无视觉信息）的$83.3\%$提升表明视觉信息至关重要，但仍有$26.7\%$的任务即使在视觉信息充分的情况下也无法完成，说明"看得到"不等于"看得懂"。

\subsubsection{缺乏真实世界知识与常识推理}

VLM的知识全部来源于训练数据中的文本和图像，缺乏对真实世界的因果理解和常识推理能力。在Web Agent场景中，这表现为：
\begin{itemize}
    \item \textbf{网站行为常识缺乏}：人类知道"点击商品图片通常会进入详情页""购物车图标通常在页面右上角"，这些常识来自大量浏览经验。VLM虽然可能在训练数据中间接学习到部分模式，但缺乏系统性的、可泛化的网站交互常识。
    \item \textbf{任务语义理解浅层化}：当任务要求"找到最便宜的手机并截图"时，人类会自然地比较多个商品的价格数值并选择最小值，而VLM倾向于选择第一个被标记的价格元素即输出answer，缺乏"比较—选择"的深层推理链路。
    \item \textbf{错误恢复缺乏因果推理}：当Agent连续两次操作无效时，VLM被prompt引导"尝试完全不同的策略"，但模型并不真正理解为什么前两次操作失败了（是因为元素不可交互、页面未加载完成、还是目标根本不在当前页面），因此"不同的策略"本质上仍是随机的探索。
\end{itemize}

\subsubsection{跨域泛化能力不足}

Mind2Web评测的三个维度（Cross-Task/Cross-Website/Cross-Domain）构成了一条泛化难度递增的曲线。Qwen3-VL-8B从Cross-Task的33.9\% Step SR下降至Cross-Domain的28.0\%，降幅达17.4\%。Qwen3.7-max同样从52.0\%下降至43.8\%。这一趋势清晰表明：模型在训练数据中见过的网站类型越多，对未见网站的泛化就越困难。

泛化能力不足的根源在于：当前VLM本质上是"插值"模型而非"外推"模型。模型在训练数据的分布范围内表现良好（插值），但面对分布外的新网站结构和交互模式时性能下降（外推失败）。这对AGI提出了一个关键问题：真正的通用智能需要在极少甚至零示例的情况下理解新环境的结构和交互规则，而当前基于统计学习的范式尚不具备该能力。

\subsubsection{多步推理与长期规划能力薄弱}

Web Agent的任务本质上是序列决策问题——Agent需要在部分可观测的环境中执行一系列操作以达到目标状态。实验数据揭示了当前模型在长期规划上的明显不足：
\begin{itemize}
    \item \textbf{规划视野短}：SoM模式下成功任务的平均步数为10.5步，但Agent在每一步的推理均以"当前页面看到什么→下一步做什么"为主，缺乏超过2-3步的前瞻性规划。当任务需要先导航到特定页面、再填写表单、最后提交的多阶段流程时，模型容易在阶段转换处迷失。
    \item \textbf{回溯能力缺失}：当Agent发现自己走错路径（如点击了错误的链接进入了无关页面），虽然prompt中提供了GOTO回退和Escape关闭弹窗的指令，但模型缺乏系统性的"状态评估→识别错误→制定恢复计划→执行恢复"能力，更多依赖prompt中的启发式规则。
    \item \textbf{探索-利用平衡差}：SoM模式平均步数（10.5步）显著高于HTML模式（6.8步），部分原因是模型在视觉模式下倾向于进行大量探索性操作（滚动、悬停、点击不同元素查看反应），缺乏高效的探索策略。
\end{itemize}

\subsubsection{缺乏持续学习与记忆机制}

当前的Agent系统对每个任务都从零开始——没有记忆、没有经验积累、无法从之前的错误中学习。在同一测试集上重复运行10次同一任务，Agent的失败模式高度一致，说明模型无法通过"试错"来改进策略。这与人类使用浏览器的模式形成鲜明对比：人类会记住常用网站的结构、积累操作经验、在不同网站间迁移相似的交互模式。

从AGI的角度看，持续学习（continual learning）和情景记忆（episodic memory）是实现通用智能的关键缺失组件。具体到Web Agent场景，理想的能力包括：
\begin{itemize}
    \item 记住特定网站的操作模式（如"QQ邮箱的撰写按钮在左侧导航栏"）
    \item 从失败中学习并避免重复相同的错误操作
    \item 将在网站A学到的交互模式迁移至结构相似的网站B
    \item 跨任务积累通用的网页操作策略
\end{itemize}

\subsubsection{效率与可靠性的根本矛盾}

在SoM自建测试集的30个任务中，平均完成步数10.5步、每步约8秒（含VLM推理3-5秒、页面加载2-3秒），总耗时约84秒/任务。人类完成相同任务的平均步数约为3-4步、每步约3秒，总耗时约12秒。效率差距约为7倍。

更重要的是，可靠性（73.3\%的任务成功率）与效率之间存在根本性矛盾：可以通过增加重试次数、降低操作节奏、增加验证步骤来提高可靠性，但会进一步降低效率；反之，减少验证和重试虽能提升速度，但错误率将显著上升。这一矛盾在当前VLM的推理范式下难以根本解决，因为模型缺乏"一次性做对"所需的深层理解和鲁棒推理能力。
